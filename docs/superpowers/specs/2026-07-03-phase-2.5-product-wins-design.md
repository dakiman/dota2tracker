# Phase 2.5 — Product wins on existing data — Design

**Date:** 2026-07-03
**Status:** Draft — pending user review (user was AFK during brainstorm; design follows
ROADMAP.md § Phase 2.5 decisions; open assumptions are flagged inline with ⚠)
**Scope:** All four Phase 2.5 items in one spec, implemented as independent tasks.
No auth, no schema surgery — one new index migration only.

## Goals

Make the site better for its actual users now, on data `player_matches` already holds:

1. **Client error handling** — distinguish 404 from 500/network everywhere.
2. **Recent matches feed** — `GET /api/matches`; home page becomes an activity feed.
3. **Played-together stats** — duo win rates + together-vs-solo, new `/together` page.
4. **Player profile page** — `/player/:id`, composed from existing endpoints.

Implementation order as numbered: error handling first (tiny, benefits every page
including the three new ones), profile last (depends on the feed endpoint).

## Non-goals

- No new tables, no auth, no rate limiting (Phase 3a).
- No "versus" (opposite-team) stats on the together page — noted as a future nicety.
- No infinite-scroll library; "Load more" button is enough.

## Architecture overview

Approach: **thin API + client composition.** Two new Hono routes (`matches.ts`,
`together.ts`); the profile page needs no new endpoint — it composes
`/api/meta?players=<id>` and `/api/matches?players=<id>`. New shared types in
`@friendtracker/shared`. One hand-written migration (0006) adding a `start_time`
index (Drizzle snapshots are out of sync — standing task — so follow the 0004/0005
idempotent-SQL convention).

A duplicated concern gets lifted while we're here: `meta.ts` and `heroes.ts` both
hand-parse/validate the `players` query param, and `matches.ts`/`together.ts` would
be copies three and four. Extract to `apps/api/src/routes/util.ts` →
`parsePlayersParam(c): string[] | null | 'invalid'` (same semantics as today:
digits-only IDs, max 50, non-empty param yielding zero valid IDs → 400).

## 1. Client error handling

**API layer** (`apps/web/src/composables/useApi.ts`):

```ts
export class ApiError extends Error {
  constructor(message: string, public status: number) { ... }
}
```

- Non-OK response → `throw new ApiError(msg, res.status)`.
- Timeout and network failures → `ApiError` with `status: 0`.
- Signature unchanged; existing `catch (e)` blocks keep working.

**UI layer:** new `apps/web/src/components/layout/ErrorState.vue` — props
`{ status: number, notFoundMessage?: string }`, emits `retry`. Renders:

- `status === 404` → the caller's not-found message (no retry button).
- anything else → "Something went wrong loading this page." + Retry button
  (re-invokes the page's `load()`).

Pages store `error: ApiError | null` instead of a string. `HeroDetailPage` passes
`notFoundMessage="Hero not found"` — a 500 or timeout no longer masquerades as a
missing hero. `MetaPage` and the three new pages use the same component.

## 2. Recent matches feed

### API — `GET /api/matches`

Query params:

- `players` — same filter semantics as `/api/meta` (via the new shared util).
  Filters **which matches appear** (any selected player participated); each card
  still shows **all** tracked participants of that match.
- `limit` — match groups per page, default 20, max 50.
- `before` — keyset cursor: epoch seconds; returns matches strictly older.
  (Tie on identical `start_time` across *different* matches is theoretically lossy
  and accepted — friend-group scale.)

Two-step query:

1. `SELECT match_id, MAX(start_time) AS start_time FROM player_matches
   [WHERE player_id IN (...)] [AND start_time < before]
   GROUP BY match_id ORDER BY 2 DESC LIMIT :limit`
2. Fetch all `player_matches` rows for those match ids (no player filter), joined
   to `heroes` (name/slug) and `players` (name/avatar); assemble groups in JS.

Response (types exported from `@friendtracker/shared`):

```ts
interface MatchParticipant {
  playerId: string; playerName: string; avatar: string | null;
  heroId: number; heroName: string; heroSlug: string;
  won: boolean; kills: number; deaths: number; assists: number; role: Role;
}
interface MatchFeedEntry {
  matchId: number; startTime: string; duration: number;
  participants: MatchParticipant[];
}
interface MatchesResponse {
  matches: MatchFeedEntry[];
  nextBefore: number | null;  // cursor for the next page, null when exhausted
}
```

`won` stays **per participant**: two tracked friends can be on opposite teams.
Same `won` value within a match ⟺ same team (one match, two teams). The card
computes its verdict client-side: all participants share `won` → single WIN/LOSS
banner; mixed → split rendering ("crossed paths" card).

### Migration 0006 (hand-written + journal entry, no snapshot)

```sql
CREATE INDEX IF NOT EXISTS player_matches_start_time_idx
  ON player_matches (start_time DESC);
```

### UI — home page becomes the feed

`HomePage.vue` keeps a compact title header, then renders the feed:

- New `apps/web/src/components/feed/MatchCard.vue` — relative time, duration,
  win/loss coloring, one row per participant (hero portrait via `heroImageUrl`,
  player name → links to `/player/:id`, K/D/A). Party games are one card.
- Respects the player-filter store like MetaPage (reload on change).
- "Load more" button drives the `before` cursor; loading/empty/error states via
  `ErrorState.vue`.

## 3. Played-together stats

### API — `GET /api/together`

Query params: `players` (optional; restricts duos to pairs **both** selected and
the per-player list to selected players).

Two aggregates, one response:

```sql
-- Duos: same team = same match + same won value
SELECT a.player_id, b.player_id,
       COUNT(*)::int AS matches,
       COUNT(*) FILTER (WHERE a.won)::int AS wins
FROM player_matches a
JOIN player_matches b
  ON a.match_id = b.match_id
 AND a.player_id < b.player_id
 AND a.won = b.won
GROUP BY a.player_id, b.player_id;

-- Per player: together (≥1 tracked same-team friend) vs solo
SELECT a.player_id,
       COUNT(*) FILTER (WHERE w.cnt > 0)::int             AS together_matches,
       COUNT(*) FILTER (WHERE w.cnt > 0 AND a.won)::int   AS together_wins,
       COUNT(*) FILTER (WHERE w.cnt = 0)::int             AS solo_matches,
       COUNT(*) FILTER (WHERE w.cnt = 0 AND a.won)::int   AS solo_wins
FROM player_matches a
JOIN LATERAL (
  SELECT COUNT(*) AS cnt FROM player_matches b
  WHERE b.match_id = a.match_id AND b.player_id <> a.player_id AND b.won = a.won
) w ON true
GROUP BY a.player_id;
```

```ts
interface DuoStat { playerA: string; playerB: string; matches: number; wins: number; winRate: number }
interface TogetherPlayerStat {
  playerId: string;
  togetherMatches: number; togetherWins: number; togetherWinRate: number;
  soloMatches: number; soloWins: number; soloWinRate: number;
}
interface TogetherResponse { duos: DuoStat[]; players: TogetherPlayerStat[] }
```

Player names/avatars come from the config store client-side (ids → players), so the
endpoint stays a pure aggregate.

### UI — new `/together` page + nav link

- **Duo table**: pair (both avatars/names), games, WR — sortable like `HeroTable`.
  Pairs under **5 games** are shown but excluded from "best/worst duo" highlights
  (top/bottom rows get a subtle badge). ⚠ threshold = 5, adjustable at review.
- **Together vs solo**: per player, two win-rate bars side by side with counts.
- Respects the player filter; `NavBar` gains a "Together" link.

## 4. Player profile page

Route `/player/:id` (`PlayerProfilePage.vue`), linked from feed cards, the together
page, and the player-filter dropdown entries. **No new API.** The page fetches:

- Player identity from the config store (unknown id → local 404 via `ErrorState`).
- `/api/meta?players=<id>` → hero-role rows. Overall WR/matches = client-side sum
  (each `player_matches` row appears in exactly one meta row, so sums are exact).
  Top heroes overall + grouped by role — reuses `HeroTable` (or a condensed variant)
  and `RoleFilterTabs`.
- `/api/matches?players=<id>&limit=10` → recent-form strip (10 W/L dots) + the
  player's recent matches, reusing `MatchCard`.

## Error handling (cross-cutting)

- All routes keep the existing try/catch → 500 JSON pattern.
- Malformed `players` → 400 (shared util); malformed `limit`/`before` → clamped /
  ignored rather than 400 (lenient on pagination noise).
- New pages ship with `ErrorState` from day one.

## Open assumptions (⚠ decided solo, cheap to change)

   share table + components).
2. Keyset `before` cursor over offset pagination for the feed.
3. Together-vs-solo defined by **same-team** tracked friends only.
4. `/together` is its own page (vs a home-page section); nav gains one link.
5. Profile page composes existing endpoints — no `/api/players/:id`.
6. Min-5-games threshold for duo best/worst highlights.
